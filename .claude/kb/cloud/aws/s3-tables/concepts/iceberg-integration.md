# Apache Iceberg Integration

> **Purpose**: How S3 Tables implements Apache Iceberg — format, REST Catalog, schema management
> **Confidence**: 0.95
> **MCP Validated**: 2026-02-19

## Overview

S3 Tables stores all data in **Apache Iceberg** format (V2, V3 supported as of Nov 2025). This provides ACID transactions, schema evolution, time travel, and partition evolution. S3 Tables also exposes Iceberg REST Catalog APIs for interoperability with any Iceberg-compatible engine.

## Apache Iceberg Fundamentals

| Feature | Description |
|---------|-------------|
| **ACID Transactions** | Atomic reads/writes via snapshot isolation |
| **Schema Evolution** | Add, rename, drop columns without rewriting data |
| **Partition Evolution** | Change partitioning without rewriting existing data |
| **Time Travel** | Query data at any previous snapshot |
| **Hidden Partitioning** | Partition transforms applied at write time, invisible to queries |

## Iceberg Table Structure in S3 Tables

```
Warehouse Location (auto-generated by S3)
├── metadata/
│   ├── v1.metadata.json     # Schema, partitions, snapshots
│   ├── v2.metadata.json
│   └── snap-{id}.avro       # Manifest lists
├── data/
│   └── *.parquet            # Data files (Parquet format)
```

S3 Tables manages the warehouse location automatically — you don't interact with raw S3 paths.

## REST Catalog APIs (March 2025)

S3 Tables exposes Iceberg REST Catalog-compatible APIs, enabling any Iceberg client to connect:

```python
# PyIceberg connection via REST Catalog
from pyiceberg.catalog import load_catalog

catalog = load_catalog(
    "s3tables",
    **{
        "type": "rest",
        "uri": "https://s3tables.us-east-1.amazonaws.com/iceberg",
        "warehouse": "arn:aws:s3tables:us-east-1:123456789012:bucket/my-bucket",
        "rest.sigv4-enabled": "true",
        "rest.signing-region": "us-east-1",
        "rest.signing-name": "s3tables",
    },
)

# List namespaces and tables
namespaces = catalog.list_namespaces()
tables = catalog.list_tables("sales")

# Load and query a table
table = catalog.load_table("sales.transactions")
scan = table.scan(row_filter="amount > 100.00")
df = scan.to_pandas()
```

## Creating Tables with Schema via Athena

```sql
-- Create table in S3 Tables via Athena (requires Glue integration)
CREATE TABLE "s3tablescatalog/my-bucket".sales.transactions (
    order_id BIGINT,
    customer_id BIGINT,
    amount DECIMAL(10,2),
    order_date DATE,
    status STRING
)
TBLPROPERTIES ('table_type' = 'ICEBERG');

-- Insert data
INSERT INTO "s3tablescatalog/my-bucket".sales.transactions
VALUES (1, 1001, 59.99, DATE '2026-02-12', 'completed');

-- Query with time travel
SELECT * FROM "s3tablescatalog/my-bucket".sales.transactions
FOR SYSTEM_TIME AS OF TIMESTAMP '2026-02-11 00:00:00';
```

## Schema Evolution

```sql
-- Add column (backward-compatible)
ALTER TABLE "s3tablescatalog/my-bucket".sales.transactions
ADD COLUMNS (shipping_address STRING);

-- Rename column
ALTER TABLE "s3tablescatalog/my-bucket".sales.transactions
CHANGE COLUMN status order_status STRING;
```

## Spark Integration (EMR/Glue)

```python
from pyspark.sql import SparkSession

spark = SparkSession.builder \
    .config("spark.sql.catalog.s3tablesbucket", "org.apache.iceberg.spark.SparkCatalog") \
    .config("spark.sql.catalog.s3tablesbucket.catalog-impl",
            "software.amazon.s3tables.iceberg.S3TablesCatalog") \
    .config("spark.sql.catalog.s3tablesbucket.warehouse",
            "arn:aws:s3tables:us-east-1:123456789012:bucket/my-bucket") \
    .getOrCreate()

# Read
df = spark.sql("SELECT * FROM s3tablesbucket.sales.transactions")

# Write
df.writeTo("s3tablesbucket.sales.transactions").append()
```

## Iceberg V3 Support (Nov 2025)

| Feature | V2 | V3 |
|---------|----|----|
| Default sort order | Yes | Yes |
| Row-level deletes | Copy-on-write | **Deletion vectors** (position deletes) |
| Variant type | No | Yes |
| Geometry type | No | Yes |
| Multi-arg transforms | No | Yes |

**Deletion Vectors** (V3): Instead of rewriting entire data files for row-level deletes, V3 uses deletion vectors that mark individual rows as deleted. This significantly improves DELETE and MERGE performance by avoiding full-file rewrites -- only a small deletion file is written alongside the data file.
