# dbt Integration Pattern

> **Purpose**: Orchestrate dbt models as first-class Dagster assets with full lineage
> **MCP Validated**: 2026-02-19

## When to Use

- Orchestrating dbt models alongside Dagster schedules/sensors
- Need cross-tool lineage (dbt models + Snowflake streams)
- Want to run dbt model subsets from Dagster UI
- Deploying dbt projects to Dagster Cloud Serverless

## Implementation — DbtProject + @dbt_assets

```python
# project.py — DbtProject with env bridging
import base64, os, tempfile
from pathlib import Path
from dagster_dbt import DbtProject
from dotenv import load_dotenv

DBT_PROJECT_DIR = Path(__file__).resolve().parent.parent / "clients" / "theraice" / "dbt_project"

# Load .env so profiles.yml env_var() calls resolve in local dev
_env_file = DBT_PROJECT_DIR / ".env"
if _env_file.exists():
    load_dotenv(_env_file, override=False)

# Cloud bridge: base64 key content → temp file → env var
if os.environ.get("DBT_SNOWFLAKE_PRIVATE_KEY_CONTENT") and not os.environ.get("DBT_SNOWFLAKE_PRIVATE_KEY_PATH"):
    _key_bytes = base64.b64decode(os.environ["DBT_SNOWFLAKE_PRIVATE_KEY_CONTENT"])
    _key_file = tempfile.NamedTemporaryFile(suffix=".p8", delete=False)
    _key_file.write(_key_bytes)
    _key_file.close()
    os.environ["DBT_SNOWFLAKE_PRIVATE_KEY_PATH"] = _key_file.name

theraice_project = DbtProject(project_dir=DBT_PROJECT_DIR)
theraice_project.prepare_if_dev()  # Runs dbt parse locally; skipped in Cloud
```

```python
# assets.py — Each dbt model becomes a Dagster asset
from dagster import AssetExecutionContext
from dagster_dbt import DbtCliResource, dbt_assets
from .project import theraice_project

@dbt_assets(manifest=theraice_project.manifest_path, name="theraice")
def theraice_dbt_assets(context: AssetExecutionContext, dbt: DbtCliResource):
    yield from dbt.cli(["build"], context=context).stream()
```

## Configuration

| Setting | Value | Notes |
|---------|-------|-------|
| `project_dir` | `clients/theraice/dbt_project` | Relative to repo root |
| `manifest_path` | Auto from `DbtProject` | Generated by `prepare_if_dev()` or CI |
| `profiles_dir` | Same as project_dir | Each client isolates its profiles.yml |

## profiles.yml — Dummy Defaults for CI/CD

```yaml
# All env_var() calls need dummy fallbacks so dbt parse succeeds without credentials
account: "{{ env_var('DBT_SNOWFLAKE_ACCOUNT', 'dummy') }}"
user: "{{ env_var('DBT_SNOWFLAKE_USER', 'dummy') }}"
private_key_path: "{{ env_var('DBT_SNOWFLAKE_PRIVATE_KEY_PATH', 'dummy.p8') }}"
private_key_passphrase: "{{ env_var('DBT_SNOWFLAKE_PRIVATE_KEY_PASSPHRASE', 'dummy') }}"
```

## CI/CD Manifest Build

```yaml
# Use dagster-dbt CLI instead of manual dbt deps + dbt parse
- name: Prepare dbt project for deployment
  run: |
    cd project-repo
    uv run dagster-dbt project prepare-and-package \
      --file dagster_orchestration/project.py
```

## pyproject.toml — Required Block

```toml
[tool.dagster]
module_name = "dagster_orchestration.definitions"
code_location_name = "dbt-orchestration-hub"
```

## Anti-Patterns

| Don't | Do |
|-------|-----|
| Run `dbt run` in a generic op | Use `@dbt_assets` for per-model lineage |
| Hardcode manifest path | Use `DbtProject.manifest_path` |
| Skip dummy defaults in profiles.yml | Add fallbacks so CI parsing works |
| Inject real secrets for dbt parse | Dummy defaults mean no secrets needed |
| Use `dbt parse` manually in CI | Use `dagster-dbt project prepare-and-package` |
