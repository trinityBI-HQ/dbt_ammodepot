# Open Data Contract Standard (ODCS)

> **Purpose**: ODCS YAML specification format, sections, and practical usage
> **MCP Validated**: 2026-02-19

## When to Use

- Need a vendor-neutral, open standard for data contracts
- Want machine-readable contracts with JSON Schema validation
- Building a contract registry across multiple teams/domains
- Require comprehensive contracts beyond just schema (SLAs, quality, ownership)

## ODCS Overview

The **Open Data Contract Standard (ODCS)** is maintained by [Bitol](https://bitol.io) under the LF AI & Data Foundation. Current version: **v3.1.2**. ODCS is now the **default format** generated by `datacontract init`, replacing the older "Data Contract Specification" format which is officially deprecated.

**Media type:** `application/odcs+yaml;version=3.1.2`

## ODCS Structure

A complete ODCS contract has 11 sections:

```yaml
# 1. Specification version
dataContractSpecification: 0.9.3

# 2. Contract identity
id: urn:datacontract:checkout:orders
info:
  title: Orders Dataset
  version: 1.0.0
  status: active
  description: "All checkout orders with line items"

# 3. Schema definition
schema:
  type: table
  fields:
    - name: order_id
      type: string
      required: true
      unique: true
      primaryKey: true
      description: "Unique order identifier (UUID)"
    - name: customer_id
      type: string
      required: true
      references: "urn:datacontract:customers:profiles.customer_id"
    - name: total_amount
      type: decimal
      required: true
      constraints:
        minimum: 0
    - name: order_date
      type: timestamp
      required: true
    - name: status
      type: string
      required: true
      enum: [pending, confirmed, shipped, delivered, cancelled]
    - name: shipping_method
      type: string
      required: false
      classification: internal

# 4. Data quality expectations
quality:
  type: SodaCL  # or Great Expectations, custom
  specification:
    - rule: row_count > 0
    - rule: "duplicate_count(order_id) == 0"
    - rule: "null_percentage(customer_id) == 0"
    - rule: "values_in(status, [pending, confirmed, shipped, delivered, cancelled])"

# 5. SLA
sla:
  freshness:
    value: 1
    unit: hour
  availability:
    percentage: 99.9
  latency:
    value: 5
    unit: minutes

# 6. Team / ownership
team:
  name: checkout-engineering
  owner: jane.doe@company.com

roles:
  - username: jane.doe
    role: Data Owner
  - username: bob.smith
    role: Data Steward

# 7. Support channels
support:
  - channel: "#checkout-data"
    type: slack
  - channel: checkout-data@company.com
    type: email

# 8. Pricing (optional)
pricing:
  priceUnit: credits
  priceAmount: 0.01
  priceCurrency: USD

# 9. Infrastructure
servers:
  production:
    type: snowflake
    account: company.us-east-1
    database: analytics
    schema: checkout
    table: orders

# 10. References
references:
  - type: documentation
    url: "https://wiki.company.com/checkout-orders"
  - type: source_code
    url: "https://github.com/company/checkout-service"

# 11. Custom properties
customProperties:
  data_classification: confidential
  retention_days: 365
  gdpr_relevant: true
```

## Validating ODCS Contracts

### JSON Schema Validation

ODCS provides a JSON Schema for IDE and CI validation:

```bash
# Validate contract against ODCS schema
pip install check-jsonschema
check-jsonschema --schemafile odcs-json-schema-latest.json contract.yaml
```

### IDE Integration

```yaml
# VS Code settings.json
"yaml.schemas": {
  "https://bitol-io.github.io/open-data-contract-standard/schema/odcs-json-schema-latest.json": "*.contract.yaml"
}
```

## ODCS vs datacontract.yaml

| Feature | ODCS | datacontract.yaml |
|---------|------|-------------------|
| Maintained by | Bitol / LF AI & Data | datacontract-cli project |
| Focus | Enterprise governance | Developer tooling |
| Schema detail | Comprehensive (logical + physical) | Practical (models + fields) |
| Quality rules | Reference-based (SodaCL, GX) | Inline checks |
| CLI tooling | Community tools | datacontract-cli (built-in) |
| Best for | Org-wide standards | Team-level adoption |

## Example Usage

```bash
# Install ODCS validator
pip install odcs-validator

# Validate
odcs validate contract.yaml

# Generate documentation
odcs docs contract.yaml --output docs/

# Check compatibility between versions
odcs diff contract-v1.yaml contract-v2.yaml
```
