# Apache XTable Interoperability

> **Purpose**: Open-source cross-format metadata translation between Apache Hudi, Apache Iceberg, and Delta Lake
> **Confidence**: 0.95
> **MCP Validated**: 2026-03-01

## Overview

Apache XTable (Incubating), formerly known as OneTable, is an open-source tool co-created by Onehouse, Google, and Microsoft that provides cross-table interoperability between Apache Hudi, Apache Iceberg, and Delta Lake. Rather than converting data files, XTable translates table metadata by reading the source format's metadata and writing equivalent metadata for target formats using each project's native APIs. This means a single copy of data files can be read by engines expecting any of the three formats.

## The Pattern

```text
Physical Data Files (Parquet in S3/GCS)
==========================================
data/
  part-00001.parquet    <-- Written once, shared by all formats
  part-00002.parquet
  ...

Metadata Layers (generated by XTable)
==========================================
.hoodie/                <-- Hudi metadata (source of truth)
  hoodie.properties
  timeline/

metadata/               <-- Iceberg metadata (translated)
  v1.metadata.json
  snap-*.avro

_delta_log/             <-- Delta metadata (translated)
  00000000.json


How XTable Sync Works:
+------------------+
| Source Format    |    read metadata
| (e.g., Hudi)    |-------------------+
+------------------+                  |
                                      v
                              +---------------+
                              | Apache XTable |
                              | Sync Process  |
                              +---------------+
                                 /         \
                write metadata  /           \  write metadata
                               v             v
                     +-----------+     +-----------+
                     | Iceberg   |     | Delta     |
                     | Metadata  |     | Metadata  |
                     +-----------+     +-----------+
```

## Quick Reference

| Feature | Hudi (Source) | Iceberg (Target) | Delta (Target) |
|---------|--------------|-------------------|----------------|
| Record-level index | Native | Via XTable metadata | Via XTable metadata |
| Time travel | Native | Supported | Supported |
| Schema evolution | Native | Synced | Synced |
| Partition evolution | Native | Synced | Synced |
| Snapshot management | Native timeline | Translated snapshots | Translated log |

| Common Use Case | Source Format | Target Format | Why |
|----------------|--------------|---------------|-----|
| Hudi writes, Snowflake reads | Hudi | Iceberg | Snowflake supports Iceberg natively |
| Hudi writes, Databricks reads | Hudi | Delta Lake | Unity Catalog prefers Delta |
| Unified lakehouse | Hudi | Both Iceberg + Delta | Maximum engine compatibility |

## Common Mistakes

### Wrong

```text
Running ETL to convert Hudi Parquet files into Iceberg Parquet files.
Maintaining separate datasets for each query engine.
Treating XTable as a data migration tool.
```

### Correct

```text
XTable translates METADATA ONLY -- it never copies or converts
data files. The same Parquet files are referenced by Hudi, Iceberg,
and Delta metadata simultaneously. XTable sync is lightweight and
can run incrementally after each commit.
```
