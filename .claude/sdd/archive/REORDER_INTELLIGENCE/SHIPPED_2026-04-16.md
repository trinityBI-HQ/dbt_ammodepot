## Summary

Added prescriptive purchasing intelligence to the Sales Dashboard — the first AI phase that tells the team *what to do*, not just what is happening. A new Gold table `F_REORDER_RECOMMENDATIONS` pre-computes per-caliber reorder quantities using ML forecast upper bounds as the safety buffer (no traditional safety-stock formula needed), identifies the best vendor by historical lead time, and estimates order cost. A new "Reorder Recommendations" tab on Page 4 surfaces this with a CORTEX.COMPLETE purchasing brief, 3 KPI metrics, and a filterable dark-themed table.

Live validation: **PASS=379, WARN=13, ERROR=0** — 103 calibers across all urgency levels. 30 Critical calibers flagged with $357K in recommended order value.

## Lessons Learned

### Technical

- **`ref()` cannot be used for Snowflake Task-managed Gold tables.** `F_FORECAST` and `F_ANOMALIES` are populated by `TASK_DAILY_FORECAST`, not dbt. Using `{{ ref('f_forecast') }}` compiles fine locally but fails at full parse in ECS with "depends on a node named X which was not found." Fix: use direct reference `ad_analytics.gold.f_forecast`. This will recur for any future dbt model that reads F_FORECAST or F_ANOMALIES — now saved to memory.

- **ML forecast UPPER_BOUND eliminates traditional safety stock formula.** Using `SUM(UPPER_BOUND)` over the 30-day forecast window gives a conservative demand estimate that naturally encodes ML-derived demand uncertainty. No Z × σ × √lead_time formula needed. Result is simpler, more accurate, and auditable.

- **No Task extension needed.** The brainstorm assumed `f_reorder_recommendations` would need wiring into `TASK_DAILY_FORECAST`. In practice, it's a standard dbt Gold table — ECS Fargate rebuilds it every 10 min automatically. Assumption A-003 was favorably invalidated during Design.

- **ECS parse failure ≠ local compilation failure.** The model passed `dbt parse` locally because partial parse skipped the bad ref. ECS does full parse from scratch every run, which caught it. Pattern to remember: always push and wait for ECS before claiming a model is valid.

### Process

- **Brainstorm surfaced the MoM constraint before build.** Asking "how do you currently use the risk table?" led to identifying that reorder *quantity* (not just timing) was the real gap. Without the brainstorm, we'd likely have built another risk view and missed the prescriptive angle.

- **YAGNI removed 6 features before any code was written.** Multi-vendor comparison, per-SKU grain, cost optimization, reorder workflow, historical tracking, overstock actions — all deferred. The MVP shipped in 1 day as a result.

- **The `ref()` bug cost one ECS cycle (~25 min).** A pre-push `dbt parse --no-partial-parse` locally would have caught it. Worth adding to the build checklist for new Gold models that reference non-dbt tables.

### Communication

- **Recommending (a) + rationale before asking** worked well. For both the quantity formula and the refresh strategy, leading with a recommendation rather than presenting options saved a round-trip.
